# 产品创意 — 字段识别：从「手写启发式」到「LLM 语义提取」混合管线

> AutoDev 阶段 1 产出 · 2026-06-08 · 源需求 `docs/PRD.md §10`（旗舰架构方向 · 提案 → 实现）
> 范围决策（PM 2026-06-08）：**§10 完整混合管线**（落地路径 1-2 步 + 为第 3 步「主路径化」铺架构）

---

## 1. 第一性原理：核心价值链分析

### 1.1 真正的痛点（一句话）

> **每换一张没见过的表单，就得给 `field-scanner.ts` 加一条 pass —— 这条「改代码」的循环是不可持续的长尾。**

证据（来自 PRD §9 迭代史，全是「dogfood → 加规则」）：

| 版本 | 真机表单 | 加的规则 | 检出变化 |
|---|---|---|---|
| V2.3 | 上海创业营 | 纯文本标签 + native 单复选 + 自定义上传 | 8→18 |
| V2.4 | 同上 | 字段顺序 + 标签隔离 | — |
| V2.5/2.6 | Epic Connector | 按钮组单/复选 + 多选 Track | +3 组 |
| V2.8 | HiCool | 两列式标签 + opacity:0 单选 | 10→15 |

根因：扫描器在**「猜布局」**（CSS class 正则 + DOM 结构规则），而不是**「理解语义」**。布局长尾无穷，规则永远追不完。

### 1.2 价值链拆解（端到端，钱/时间从哪儿省）

```
用户痛点链：
  遇到新表单家族  →  扫描器漏检/错标  →  PM 手动补字段/纠正  →  价值打折
        ↓ 现状解法（不可持续）              ↓ 本方案解法（可持续）
  开发者加一条 pass（改代码、发版、回归）   LLM 天生「见过海量表单」，直接理解控件语义
```

**核心洞察**：LLM 不需要 class 规则就懂「这个框在问什么」。把识别从「规则匹配」换成「语义理解」，**识别类缺口换表单零代码改动**。

### 1.3 致命技术坑 + 破解（为什么以前没人这么做）

| 坑 | 说明 | 破解 |
|---|---|---|
| **LLM 给不出能写值的选择器** | 纯视觉控件（styled `<button>`、opacity:0 input）LLM 描述得出、却对不回具体 `<input>`，无法回填 | **`data-af-id` 打标**：扫描前给每个候选控件挂稳定 id，LLM 只返回 afId，再精确对回 DOM。这是整个方案成立的支点。 |
| **LLM 幻觉 / 漏检 / 错标硬约束** | 模型会编不存在的字段、漏掉真字段、把 maxLength 猜错 | **DOM 是硬约束唯一真相**（maxLength/options/accept 信 DOM 不信 LLM）+ afId 存在性校验剔除幻觉 + 启发式交叉校验兜底漏检 |
| **时间类字段扫描时不在 DOM** | 条件展开/分页/重复行的字段，静态扫描原理上看不见 | **诚实边界**：静态方法（规则或 LLM）都看不见，不承诺。留给交互式探索（V2.7「继续下一页」已开头，未来 phase 2） |

## 2. 诚实边界（产品价值的护栏，不可越界）

PRD §10 已明确，本节固化为铁律：

- ❌ **不承诺「无人复核、任意表单 100% 一一对应」** —— LLM 会错、时间类要交互，这是做不到也不该承诺的。
- ✅ **真正消灭的是「换网页就要改代码」** —— 这是用户真正想要的。
- ✅ **守住「提交前人工复核闸」** —— 与 PRD「明确不做 Auto-submit / 用户必须 review」一致。
- 「100% 一一对应」拆成两类缺口：

| 缺口类 | 是什么 | 本方案可解性 |
|---|---|---|
| **识别类**（机器零件混入 / N 控件=1 字段 / 1 模板=N 字段 / 无声明式标签） | 字段已在 DOM，难在判断「哪些是字段、问什么、怎么分组」 | ✅ **可解到「换表单不改代码」**（本次 MVP 目标） |
| **时间类**（条件展开 / 加队员重复行 / 分页） | 字段扫描时还不在 DOM | ⏳ 静态方法看不见，只能靠交互逼出（不在本次范围） |

## 3. MVP 功能清单（P 平台层 / W 场景层）

> 编号约定：P = 底层平台能力（管线零件），W = 面向用户的场景能力。

### 3.1 平台层（P）—— 混合管线五件套 + 支撑

| 编号 | 能力 | 描述 | MVP |
|---|---|---|---|
| **P1** | DOM 打标引擎 | 遍历 DOM，给每个可能可交互元素（input/select/textarea/`[role]`/contenteditable/styled-clickable）挂 `data-af-id`；幂等、可重入 | ✅ 必须 |
| **P2** | 控件清单蒸馏 | 生成**精简清单**（afId + tag/type/placeholder/附近可见文字/可见性 + DOM 硬约束）；仅可见元素、控体积；**不发原始 HTML** | ✅ 必须 |
| **P3** | LLM 语义提取 | 一次调用，返回「人会填的字段」（label / type / 约束 / 是否敏感 otp\|personal / 多控件合一）；排除后端/隐藏/动作控件。**复用 `callLLM` + `detectEventFromPage` 模式** | ✅ 必须 |
| **P4** | afId 回填 + 校验 | afId 精确对回 DOM 节点（解决纯视觉对不上）；硬约束信 DOM 不信 LLM；校验 afId 真存在（防幻觉）；产出标准 `DetectedField[]` | ✅ 必须 |
| **P5** | 扫描结果缓存 | 按 URL + DOM 签名缓存，重复访问不重复调用 LLM | ✅ 必须 |
| **P6** | 扫描模式开关（feature flag） | `heuristic`（现状）/ `hybrid`（默认新）/ `llm`（纯 LLM）三态；默认仍走启发式，新路径 flag 后 | ✅ 必须 |
| **P7** | dogfood 语料 + recall 基准 | 把已 dogfood 的表单（科大硅谷/HiCool/Epic Connector/上海创业营）存成离线 HTML fixture，量 recall（检出/真实字段），做回归基准 | ✅ 必须 |

### 3.2 场景层（W）—— 用户能感知的

| 编号 | 能力 | 描述 | MVP |
|---|---|---|---|
| **W1** | 混合扫描 | heuristic 先跑（离线快路径）→ LLM 对差异/补漏（completeness-repair pass）→ 合并去重。比从零提取更省更准 | ✅ 必须 |
| **W2** | 纯 LLM 扫描模式 | 完全靠 LLM 语义识别，**换没见过的表单一行代码不改**（landing-path 第 3 步「主路径」的能力，本次架构就绪、flag 可选开） | ✅ 架构就绪 |
| **W3** | 字段来源透明化 | provenance 新增 `llm-semantic` 来源；FieldExplainer 显示「LLM 识别 / 启发式识别 / 两者一致」 | ✅ 必须 |
| **W4** | recall 对比可视 | 扫描后显示 heuristic vs hybrid 检出数对比（让 PM 看见「LLM 多捞了几个」） | 🟡 增强 |

### 3.3 明确不在本次范围（守边界）

- ⏳ **时间类交互式探索**（点「有公司」展开、点「加队员」materialize 重复行、自动翻页）—— 会改动表单状态，需独立的用户监督/可回退设计，留 phase 2。
- ❌ 真 embedding RAG、Auto-submit、多语言 UI、Firefox 适配 —— PRD §10 其他候选，与本旗舰方向无关。

## 4. 失败模式分析（Q3：这个产品会怎么死）

> 「不预演失败模式 = 自导自演降阶」。逐条列出 + 缓解，缓解措施直接进 P/W 验收标准。

| # | 失败模式 | 后果 | 缓解（落到哪个能力） |
|---|---|---|---|
| F1 | LLM 幻觉 afId（返回不存在的 id） | 回填崩 / 写错框 | P4 afId 存在性校验，不存在直接剔除 |
| F2 | LLM 漏标真实字段 | recall 反而比启发式低 | W1 hybrid：启发式交叉校验兜底；P7 recall 基准量化把关，**LLM 稳超启发式才设主路径** |
| F3 | LLM 把后端/隐藏/动作控件当字段（HiCool 实测 48 控件→16 可填） | 噪声字段、误填 | P2 蒸馏只发可见元素；P3 prompt 显式要求排除；P4 回填再校验可见性 |
| F4 | 纯视觉控件 LLM 对不回 `<input>` | 识别到却填不了 | **P1 data-af-id**（方案支点，专治此症） |
| F5 | LLM 把 maxLength/options/accept 猜错 | 草稿超限、选项错 | P4 硬约束信 DOM 不信 LLM（DOM 抽真值覆盖 LLM 值） |
| F6 | 每次扫描都调 LLM → 成本/延迟爆炸 | 慢 + 烧 token | P5 缓存（URL+DOM 签名）；P2 控清单体积；扫描本不频繁 |
| F7 | 控件清单含敏感页面数据发给 LLM | 隐私顾虑 | 与 `detectEventFromPage` 已发页面正文**姿态一致**；P3 默认只发可见控件元信息（非完整 HTML）；后续可加站点白/黑名单 |
| F8 | LLM 不可用 / 超时 / 429 | 扫描整体失败 | W1 启发式是**离线兜底**：LLM 失败自动退回纯启发式结果，绝不白屏 |
| F9 | 时间类字段（条件/分页/重复行）被当「已全」 | 用户以为扫全了其实漏 | F9 不假装能解：UI 明示「本页静态字段」+ 复用 V2.7「继续下一页」；**不在文档/UI 声称扫到了动态字段** |
| F10 | 把 AX 无障碍树当唯一输入 | HiCool 实测漏 opacity:0 单选 + 自定义下拉 | P1/P2 用**原始 DOM** 打标蒸馏，不依赖 AX 树（PRD §10 已记取的数据点） |
| F11 | 推倒启发式重来，回归风险高 | 旧表单反而变差 | **不推倒**：启发式保留为快路径 + 交叉校验 + LLM 失败兜底；LLM 收窄为「补差异」 |

## 5. 与现有架构的契合点（为什么这个方案低风险）

- **复用 `detectEventFromPage`（background.ts:296）模式**：它**已经在把页面正文喂 LLM 提取活动信息**，字段识别就是同一套模式延伸到控件 —— 不是新引入「页面发 LLM」的姿态。
- **复用 `callLLM`（client.ts:337）+ `parseBatchResponse`/`structuredExtract`（prompts.ts）**：LLM 调用 + 容错 JSON 解析的轮子都现成。
- **产出标准 `DetectedField[]`**：与启发式同形，下游 draft 生成 / 一键填入 / provenance UI **全部不动**。`domSelector` 用 `[data-af-id="..."]`（自己挂的、最稳）。
- **不破坏耐久铁律**：图标 lucide（W3/W4 UI）、`exactOptionalPropertyTypes`、改完回写 3 处文档。

## 6. 成功的样子（验收时回头看这里）

1. 把一张**没在代码里特判过**的真实表单喂进去，hybrid/llm 模式检出 recall ≥ 纯启发式，**且没改一行 scanner 规则**。
2. 旧的 4 张 dogfood 表单（语料库）recall 不退化（启发式兜底保证）。
3. LLM 失败时自动退回启发式，不白屏。
4. 硬约束（maxLength/options）始终来自 DOM，从不被 LLM 污染。
5. 重复访问同页不重复调 LLM（缓存命中）。
6. UI 诚实：动态字段不谎称扫到；字段来源可在 FieldExplainer 里看到是 LLM 还是启发式。
