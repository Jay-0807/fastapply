# 01.5 — User Research: V1 已发现的交互痛点

> 不重做用户研究，而是把"已知的 30+ bug 报告 + 代码层 22 项审计"按用户旅程结构化，定位**根因桶**，从而让 05a 的方案设计有的放矢。

## 一、用户画像（在 V1 之后更新）

**主画像**：非技术 PM，jayyangstudy@gmail.com
- 已**安装并使用过**插件，跑过创投/黑客松类报名（含 Qualtrics、Google Forms）
- 已配置 Anthropic API key + master password
- 已建立至少 1 个项目档案 + 上传文档 + 上传资产
- **当前心智**：「这堆功能我都见过了，但每次用都很卡，不知道在哪一步会爆什么问题」

**典型一次使用旅程**（基于 PM 反馈复盘）：
1. 在浏览器打开报名页（Qualtrics / Google Forms）
2. 点扩展图标 → 打开 sidepanel
3. 选项目 → 下一步
4. 看到事件信息（活动名/主办方）—— **这里经常是错的或空的**
5. 改一下，点"确认并扫描字段"
6. 看到 N 个字段（**N 可能 = 真实题数，也可能 57 个全是噪声**）
7. 点"AI 生成全部" → 等待（**有时全部生成完，有时卡在第 3 个不动**）
8. 看草稿，改文字
9. 点"一键填入页面" → **要么填好了，要么报错没下文**
10. 浏览器里手动检查 + 补漏
11. 提交真实表单
12. 回来点"我已提交，沉淀经验" → **要么成功下载 markdown，要么报错没下文**

### Bucket 3 — 抓不准内容

| 优先级 | 文件 / 位置 | 问题 | 用户感受 |
|---|---|---|---|
| **P0** | `field-scanner.ts:403-406` `docHasLayout` gate | 真实 Chrome 0×0 文件输入逃过可见性检查 | 文件字段时有时无（同上） |
| **P1** | `field-scanner.ts:53-69` `MAX_LENGTH_PATTERNS` 不全 | 漏了"限 X 字 / 约 X 字 / 控制在 X 字" | AI 生成超字数被截断 |
| **P1** | `field-scanner.ts:355-365` file input 旁路可见性检查 | 选不到真正的隐藏 drop zone 字段 | 上传字段缺失或填错位置 |
| **P1** | `field-scanner.ts:231-240` ARIA group 无 label 静默跳过 | 没 aria-label 的单选组直接消失 | 用户看到题数变少不知道 |
| **P1** | `field-scanner.ts:39-51` `ADMIN_LABEL_PATTERNS` 仅覆盖 Google Forms | Jotform / Gravity Forms 等表单编辑器没拦截 | 在表单编辑页打开会扫到 N 倍噪声 |
| **P1** | `background.ts:268-326` body 只取前 4000 字 | 长页面活动信息在 5000 字后被切掉 | "New Application Form" 这种 hallucination |
| **P1** | `field-scanner.ts:191-254` ARIA 组不递归 Shadow DOM | LWC / Shoelace / Web Components 字段抓不到 | 整组单选/复选凭空消失 |
| **P2** | `field-scanner.ts:270-315` `detectGroupLabel()` 仅向上 4 层 | 嵌套深的现代框架找不到 heading | 同上 |
| **P2** | `field-scanner.ts:827-846` `clickMatchingOption()` 子串匹配过宽 | "Option A" 误点中 "Option AB" | 选错答案 |
| **P2** | `field-scanner.ts:628-635` helper text 检测漏 `<small>` / `text-muted-foreground` | 提示语没读到，AI 不知道字段约束 | 生成内容跑题 |

**合计**：3 桶 22 项（P0×2 / P1×15 / P2×5）。

## 三、根因桶映射 → 设计原则

| 根因桶 | 共性结构问题 | 设计原则（→ 给 05a 用） |
|---|---|---|
| **B1 看不见进展** | 异步操作没有统一的 UX 包装；catch 错误只走 console；fallback 路径用户不知 | **原则 1**：所有异步入口必须显式三态（idle / busy / done-or-failed）+ 错误必须有用户可读消息 + 兜底必须显式提示 |
| **B2 记不住设置** | 大量"应该是 session 级"的状态被放在 React useState 里；persist 时机不严谨 | **原则 2**：区分三层状态（`localState`/`tabSession`/`projectSession`），按层落地（useState / chrome.storage.session / db.appSettings） |
| **B3 抓不准内容** | scanner 的多个 heuristic 彼此打架；fallback 无声 | **原则 3**：scanner 必须能解释自己（每个字段告诉用户为什么被纳入，每个跳过告诉用户为什么没纳入）；提取必须有置信度而不是空字符串 |

**额外**：原则 4（Karpathy "Goal-Driven Execution"）—— 重设计后每个操作有可验证的成功标准（不是"看起来在转圈"），原则 5 ——优先级修 P0 ≫ P1 ≫ P2，本迭代承诺 17 项（P0×2 + P1×15），P2 在新一轮做。

## 四、要保留的 6 个好 UX 模式（不要回归）

来自上一轮迭代修过的、当前已经做对的：

1. **`LockedFieldsCard` 内联解锁** —— "Settings locked" 错误就地解锁，不弹窗、不踢出流程 → **保留 + 推广**
2. **`fieldState` 四态机** —— ⏳ / ✏️ / ✅ / ❌ 状态徽章 → **保留 + 推广到所有异步操作**
3. **`batchProgress` 序列进度** —— "正在生成第 X / Y 个…" → **保留**
4. **120s 显式超时 + "生成超时（120s 无响应）"提示** → **保留 + 推广到 fillPage / scan / detect**
5. **扫描失败时的 contextual hint**（编辑页 URL 检测后给提示） → **保留 + 扩展更多场景**
6. **`generateOneDraft` 同时 broadcast + re-throw 双兜底** → **保留**

## 五、本迭代不修的（推下一轮 / 标记不影响）

- B4 "搞不清产品"桶全部（onboarding / 三入口职责 / 备份-经验库的产品语言）—— PM 暂未勾选
- P2 项中的：scanner heuristic 收紧（`clickMatchingOption` 子串、helper text 多源检测、Shadow DOM 递归）—— P0+P1 修完再做
- 新功能 / 新数据源 —— 范围外
