## 关键洞察

### 1. 我们其实**从未跑通一次 step 10**

到目前为止所有 bug 都在 step 1-9。生成草稿这步**还没有任何一次真正成功**。继续 reactive 修复只会让用户继续放弃。

### 2. 真正的"端到端验证"必须包括 step 10-12

只有用户**真的看到一份 markdown 文件生成、history 里多一条记录**，才算这个产品有了一次"完整闭环"。否则前面所有功能都是花架子。

### 3. 用户卡住时缺乏自救路径

当生成失败时：
- 当前现状：用户只看到"生成中..."永不结束
- 用户希望：看到具体错误 + 一键修复按钮（"看起来你的 session 锁定了，输主密码解锁？"）
- 这是 P0 的真正痛点 — **不是修底层 bug 就够，要修反馈链**

### 4. 大部分"已修"是技术修复，没修用户体验

| 已修的技术 bug | 用户视角的对应改进 |
|---------------|------------------|
| sessionKey 持久化 | ❌ 用户还是要"先去设置再保存一次" — 需要自动 migration 提示 |
| sendBg envelope 解包 | ❌ Side panel 还用旧的 sendMessage helper（line 257）— **同一个坑沒一起修** |
| CSS 全局加载 | ✅ 用户能看到了 |
| EventField hoist | ✅ 用户能正常输入了 |
| 删除/重试按钮 | ✅ 用户能自救了 |

## 反方观点 / 风险

1. **会不会还有更深的雷在 step 11-12**：可能。Google Forms 是 React-rendered，input/change 事件可能不被 React 监听（要 React-specific 触发）。**这是 P0-3 修复时必须验证的，不能纸上谈兵。**
2. **Claude API 调用本身会不会失败**：可能。网络、限流、key 失效都可能。P0-1 的"显示错误并自救"必须包含这些场景。
3. **3.5h 估算偏乐观**：是的。如果 Google Forms 兼容性问题大，可能要追加 2-3h。**承诺只在 P0 完成后再开新 issue，不无限延展。**
