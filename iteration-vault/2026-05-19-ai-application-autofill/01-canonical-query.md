## 问题陈述（What & Why）

PM（用户）需要长期参与各类项目报名：黑客松、创投活动、课程报名、政策申请、路演、其他赛事。每次都要在不同平台手动填一份长表单，**同一个项目的同一个介绍要根据不同问法写很多遍**，并且**根据不同活动主题/主办方/地区产业特色微调措辞才能提高入选率**。

**当前痛点**（已实测验证）：
1. ❌ 手填：每次 1-2 小时
2. ❌ 一个个问大模型：上下文反复粘贴、回答风格漂移
3. ❌ Claude for Chrome（实测）：
   - 信息填入有误
   - 鼠标移动慢、效率低
   - **看不到字段约束**（字数限制 / placeholder hint / required 等）
   - 每次要重新喂资料，不积累
   - 价格高（$20/月起）

**做了能衡量的好处**：
- 单次填表时间：1-2 小时 → 10-15 分钟（**5-8 倍提速**）
- 入选率：因更匹配活动背景而提升（不可精确量化，靠主观判断）
- 长期积累：每次申报形成可复用资产（Q&A markdown），AI 回答越用越收敛

## 解决方案高层描述（How）

一个 Chrome 浏览器插件（**Manifest V3**），具备 3 个核心能力闭环：

### Loop 1 — 智能填表（每次报名的当下价值）
1. 检测到报名页面 → 弹出"准备填表"面板
2. PM 选择/创建本次报名所属的【项目】（如"firefly OS"）和【活动背景】（主题/地点/主办方/链接）
3. AI 从【项目档案】+【历史 Q&A 经验库】+【活动背景】综合生成草稿答案
4. **关键差异化**：插件解析每个字段的完整约束（label / placeholder / maxlength / required / helper text / aria-describedby），让 AI 生成"符合长度限制 + 切题"的答案
5. PM 审阅 → 一键填入或逐字段修改

### Loop 2 — 经验沉淀（每次报名后的资产积累）
1. 提交后，插件自动把【活动背景 + 每个 Q&A + PM 的最终修改】保存为一份 markdown 文件
2. 这份 markdown 自动进入【历史经验库】，作为下次填类似字段的 RAG 上下文
3. **越用越收敛**：第 1 次报黑客松要花 30 分钟修 AI 草稿，第 10 次只要 5 分钟（AI 已经学会了 PM 的偏好风格）

### Loop 3 — 活动匹配增强（提高入选率）
1. PM 填【活动背景】时，插件 AI 推断主题/地点/主办方草稿，PM 30 秒确认
2. 生成 Q&A 时，AI 把活动背景作为强 context：
   - "上海主办的智能制造活动" → 答案突出长三角产业链 + 智能制造维度
   - "深圳硬件加速器" → 答案突出供应链 + 量产经验
   - "杭州 AI 创投" → 答案突出 AI 技术栈 + 大模型应用
3. 经验库里同一个问题在不同活动背景下会有多版回答，AI 选最匹配的复用

## 范围（Scope - In）

V1 必须有：
- ✅ Chrome Extension MV3 框架（WXT/Plasmo 选其一）
- ✅ 项目档案 CRUD（创建/编辑/删除 1 个或多个项目，上传 PDF/Word/MD/Txt 文档）
- ✅ 活动背景输入（AI 推断 + PM 确认）
- ✅ 表单字段识别（含约束解析：label/placeholder/maxlength/required/helper text）
- ✅ AI 生成草稿（基于项目档案 + 历史 Q&A + 活动背景）
- ✅ 一键填入页面 + 逐字段手动覆盖
- ✅ 报名后自动生成 Q&A markdown
- ✅ 历史经验库 RAG 检索（IndexedDB 本地向量存储）
- ✅ Claude API 接入（Sonnet 主用 + Haiku 备用）
- ✅ API key 本地加密存储（chrome.storage.local）
- ✅ 中英文双语字段识别

---

## 范围排除（Scope - Out / Not in V1）

明确不做：
- ❌ 上 Chrome Web Store（自用版本）
- ❌ 账号体系 / 多用户隔离
- ❌ 云端同步（仅本地 IndexedDB）
- ❌ 其他 LLM 提供商（GPT/DeepSeek/通义 — 仅 Claude）
- ❌ 移动端（Chrome 桌面版 only）
- ❌ Firefox / Edge 兼容（V1 仅 Chrome）
- ❌ 多步表单/复杂条件跳转的自动化（V1 仅"看到啥填啥"，不做 multi-step workflow）
- ❌ 验证码识别 / 文件上传自动化
- ❌ 真正的"自动提交"（V1 始终需 PM 点提交，避免误提交）
- ❌ PDF 形式的报名表（仅 Web 表单）

## 关键非功能性需求

| 维度 | 要求 | 备注 |
|------|------|------|
| 性能 | 单字段 AI 生成 < 5s（Haiku）/ < 15s（Sonnet） | 用 streaming 给 PM 即时反馈 |
| 隐私 | 所有数据本地存储，**只有调 Claude API 时数据出本地** | 用户的项目资料是商业敏感信息 |
| 安全 | API key 加密存 chrome.storage.local | 不存 plain text |
| 可用性 | 即使 LLM API 失败，也能进入"手动填表 + 保存 Q&A"模式 | 不能因 API 挂了就完全用不了 |
| 可访问性 | UI 中英文双语 | i18n 框架预留 |

## 验收标准（V1 Done 的二元清单）

PM 在 1-2 次真实活动报名中能验证：

- [ ] 能创建至少 1 个项目档案，上传 ≥3 份相关文档（PDF/Word/MD）
- [ ] 在 ≥2 个不同报名平台（如金数据 + Devpost）打开插件能正常识别 ≥80% 字段
- [ ] AI 草稿包含了项目档案里的关键信息（不是凭空生成）
- [ ] 字段约束生效：标记了 "100 字以内" 的字段，AI 生成的内容不超过 100 字
- [ ] 活动背景能影响草稿（同一字段用不同活动背景生成会有差异）
- [ ] 报名完成后能在本地看到 `<活动名>-<日期>.md` 文件，含完整 Q&A
- [ ] 第 2 次填类似字段时，AI 能引用历史经验库（可在 UI 里看到"参考了上次 XX 活动的回答"）
- [ ] 全程不需要 PM 主动复制粘贴任何文字到外部 LLM 工具

## 测量指标（上线后怎么衡量好坏）

V1 自用阶段（仅 PM 一人），主观指标为主：

| 指标 | 测量方式 | 目标 |
|------|---------|------|
| 单次报名耗时 | 自己计时 | 从 1-2h 降到 < 20min |
| AI 草稿采纳率 | 每个字段：直接用 / 微改 / 重写。记日志 | 直接用 + 微改 ≥ 70% |
| 经验库收敛度 | 第 N 次填类似字段时，AI 草稿的"采纳率"是否随 N 上升 | 第 10 次相比第 1 次 +20% 采纳率 |
| 主观满意度 | 用满 1 个月后自评 | 比"不用插件"满意度 ≥ 8/10 |

## 关键的"差异化设计"备忘

这些是本插件区别于现有所有方案的核心点，PRD 阶段要强调：

1. **字段约束感知** — Claude for Chrome 失败的关键点，必须解决
2. **活动背景驱动** — 同一问题在不同活动背景下生成不同答案
3. **历史 Q&A 经验库** — 越用越好，可持续资产
4. **本地优先 + Markdown 导出** — 隐私 + 数据可移植
5. **PM 始终在 loop** — 不自动提交、不替 PM 决策，AI 是助手不是 agent
